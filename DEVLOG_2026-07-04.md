# The landmine: `scenarios/_llm.py` still depended on Streamlit

The shared OpenAI client that `server.py` and `interpretation.py` actually use
in production still did `import streamlit as st` and fell back to
`st.secrets.get("OPENAI_API_KEY")`. Removing `streamlit` from
`requirements.txt` without fixing this would have turned into a hard
`ImportError` on every production request. Fixed: `_get_api_key()` now just
does `os.getenv("OPENAI_API_KEY")`; dropped the Streamlit-specific docstrings
and log/error text ("Streamlit Cloud logs", "current Streamlit run",
"Streamlit secrets"). The lazy re-resolve-on-key-change behavior in
`get_client()` was kept ‚Äî still useful for a long-lived process, nothing to
do with Streamlit specifically.

Also dropped `streamlit==1.57.0` from `requirements.txt`, and cleaned up a
dangling comment in `aam.py` that pointed at
`_derive_noi_from_pricing in aam_extractor` (function no longer exists).

# Where the live API key goes (for next time someone tests the `source=gpt`
path end-to-end ‚Äî open item from 2026-06-30)

`scenarios/_llm.py::_get_api_key()` now reads only `OPENAI_API_KEY` from the
environment.

- **Local**: `python-dotenv` isn't wired up anywhere in this repo, so a
  `.env` file won't auto-load ‚Äî export it in the shell: `export
  OPENAI_API_KEY="sk-..."` before `uvicorn server:app --reload`.
- **Cloud Run (`paw-prints-api`)**: set at deploy time, e.g. `gcloud run
  deploy paw-prints-api --set-env-vars OPENAI_API_KEY=...` ‚Äî plain env var,
  no Secret Manager configured. Deploy is manual from desktop; `git push`
  alone does not update the live service.
