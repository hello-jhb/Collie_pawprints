# Spec 1 — represent the rate lever; stop assuming occupancy

The Read hard-coded occupancy as the value lever: `_classify_archetype` and the
`thesis` claim only read occupancy + capex + NOI ramp, and System Prompt Rule 5
literally asserted "occupancy is the lever behind the ramp." No ADR/RevPAR series
existed anywhere — `cashflow_rollup` discarded ADR as "not summable."

**ADR/RevPAR as a level series.** Added `adr`/`revpar` to `workbook_map`'s
concept vocabulary with a `$10–$100,000` dollar-level domain gate, and
`cashflow_rollup.rate_level_candidates()` mirroring the existing
`occupancy_candidates()` — a $ *rate* per key/night, averaged per year, never
summed (divides the sheet's declared units back out the same way occupancy does).
Generalized the occupancy plumbing in `deal_analysis.py` into shared
`_select_level_series` / `_level_bookends` reused by occupancy, ADR, and RevPAR.
`_forward_year_avg` gained an `anchor_bp` param so a rate series that carries
years of history before the NOI stream starts still reads its going-in window
relative to *NOI's* start, and now handles annual-grain series (one point/year)
as well as monthly.

**RevPAR bridge.** New `_revpar_bridge()` splits the RevPAR change from going-in
to stabilized into the portion driven by ADR vs. occupancy (RevPAR = ADR ×
Occupancy, exact average-rate/average-volume attribution, cross-term split evenly
so the two effects sum with no residual) and names the dominant `lever`
(`rate` / `occupancy` / `rate + occupancy` / `flat`). On St. Regis this reads
`rate_share ≈ 1.29, occ_share ≈ −0.29` — occupancy is actively working *against*
RevPAR growth, exactly the spec's diagnosis.

**Lever-aware thesis.** The `thesis` claim now names the actual lever via
`_lever_phrase_and_guardrail()` instead of assuming occupancy, and emits a
guardrail (`"The value lever is RATE (ADR); occupancy is down. Do not attribute
the ramp to occupancy."`). Relaxed Rule 5 to "tie the NOI trend to whichever
operating lever is actually moving."

# Spec 1 — distinguish renovation disruption from lease-up (P3), fix the anchor (P2)

Any early NOI trough was treated as lease-up (expected, ignorable). St. Regis is
an operating asset taken *offline to renovate* — same low DSCR, different risk.

**Reposition detector.** New `_detect_reposition()` fires on a mid-hold
**V-shaped NOI trough** (healthy → deep dip → recovery to a *new higher* plateau)
with **capex concentrated in the trough** and **non-trivial going-in occupancy**
(the asset was operating) — distinguished from a lease-up that ramps monotonically
from ~zero. It trims trailing structural-zero years and only considers near-full
(≥11-month) years so a partial close-stub can't masquerade as the trough. New
`"value-add / repositioning"` archetype + lens; `_classify_archetype` overrides to
it when the pattern fires; the `structural_risk` claim labels the low-coverage
window **"renovation disruption"** instead of "lease-up."

**Going-in NOI anchor (P2).** `_clean_going_in_noi()` steps the going-in NOI off
a partial close-stub or the renovation trough to the nearest clean, undisrupted
full year — gated on the reposition flag so an ordinary lease-up deal (where the
rolling window was deliberately chosen over calendar years) is untouched. On
St. Regis this fixed going-in NOI from the old stub-year `$7.5M` to `$8.05M`.

*(Note: the spec's illustrative "≈$12.3M in-place 2018" doesn't cleanly trace to
any single row in the actual workbook — I reported the value that does rather than
force the number.)*

# Spec 2 — state the deal phasing

The Read could classify a deal but couldn't state its *timeline*. New
`_deal_phasing()` derives, from the dated NOI + capex series alone:
build/reno window (+ its capex share) → delivery/reopen → lease-up →
stabilization → post-stab hold. `{"kind": "none"}` for a stabilized deal.

- **Repositioning** window = the rooms-dark offline run around the trough,
  anchored to the reposition's `trough_year` (a series that opens with zero months
  at close would otherwise tie the "min" onto the series start).
- **Development** window = close → first material-NOI month, only when the gap
  is ≥12 months.
- **Stabilization** reuses `_dscr_health`'s `stabilization_month` when present,
  else a new `_ttm_stabilization_month()` applying the *same* ≥95%-of-stabilized
  TTM rule to NOI alone (verified it reproduces `2023-08` exactly — one
  definition, not a competing one).

Carried on the fact sheet (`deal["phasing"]`), rendered as a `PHASING:` line, and
narrated by a `phasing` claim whose guardrail forbids inventing or rounding dates.
Prompt Rule 7b lets the read open the thesis with the timeline. On St. Regis:

> ≈15-mo renovation (93% of capex) · reopen 2021-07 · stabilize 2023-08 · 4 mo
> post-stab hold  (T2)

The spec's illustrative "18–24 month" reno is approximate; the model's actual
offline window is a clean 15 months (rooms at zero NOI Apr-2020 → Jun-2021), so I
reported the true figure. The short 4-month post-stab hold is also genuine — the
model's NOI ends at exit shortly after stabilization.

# The real bug this surfaced: bare-year headers hijacking `find_spine`

St. Regis's ADR/Occupancy rows live on a `Proforma` tab keyed off **text-year**
headers (`"2014"`, `"2015"`…), which `cashflow_spine._parse_textdate` didn't
recognize (only *integer* years). Adding bare-year-string parsing made those rows
visible to the rollup — but it also widened `find_spine`'s candidate-stream search
(a far higher-stakes surface), and a comp-set/scenario tab's text-year column
briefly out-competed the real engine: `total_cost`, `debt`, and `sale_price` all
jumped onto the wrong sheet (`total_cost` read `$76`, `debt` read `$241K`).

Caught it by diffing the canonical stack against the known-good baseline before
trusting the fact sheet. Fixed by making bare-year-string parsing **opt-in**
(`allow_bare_year_text`, threaded through `_coerce_date` / `_date_run`): only the
rollup's level-series axis detection opts in; the spine's own stream search leaves
it off. Baseline stack restored (`$371.8M` / `$240.9M` / `$526.2M`). Also
extended the source-conflict guardrail suppression to `adr`/`revpar` so a stray
market-comp cell doesn't fight the correctly-selected series.

# Follow-up (same thread): ADR leaked into a multifamily

The user uploaded `1425 4th Ave MF Proforma` — a **212-unit multifamily** — and
the Read called it *"a rate-led value creation story... ADR moving from $271 to
$348."* The model carries an ADR/RevPAR row for a short-term-rental ("Mint House")
*sub-component*; the new feature picked it up as the *deal-level* rate lever. ADR
is a hotel metric — a category error for a multifamily.

- `build_analysis` now gates the ADR/RevPAR read on `property_id` classifying the
  asset as hospitality (type hotel/hospitality/resort/lodging, or size in
  keys/rooms). Non-hotels never populate `adr`/`revpar`/`revpar_bridge`, so the
  archetype lever, thesis headline, and fact sheet all inherit the gate. Property
  identity is now computed once in `build_analysis` and returned for the
  interpretation layer to reuse (no second workbook read).
- `_lever_phrase_and_guardrail` no longer names occupancy as the lever when
  occupancy is *flat* (<3% move) and there's no rate series — a flat-occupancy NOI
  ramp is a rent-growth story we can't cleanly measure, not an occupancy story.

The gate is deliberately conservative: if property-type detection can't confirm a
hotel, ADR stays suppressed. Both local models detect cleanly (Hotel vs.
Multifamily), so neither regresses. 1425 now reads `Value-Add — NOI $3.3M → $7.7M
(+132%); occupancy 70% → 70%` with no ADR and no false lever; St. Regis is
unchanged.
